String、StringBuffer和StringBuilder

　　String是一个不可变的对象。就是说String本身一经创建则不可改变，每次对String内容的改变实质上都是创建了一个新的String对象。
　　StringBuffer是一个可变的对象，对于StringBuffer的改变并不会创建新的对象。同时，StringBuffer还是一个线程安全的对象。在多线程并发情况下，使用StringBuffer可以得到正确的结果。
　　StringBuilder是在JDK1.5中引入的。它与StringBuffer类似，但是它并不是一个线程安全的对象。

1. String与StringBuffer

　　String与StringBuffer的源码如下： 

//String   
public final class String  
{  
    private final char value[];  
    public String(String original) {  
      // 把原字符串original切分成字符数组并赋给value[];  
    }  
}  

//StringBuffer   
public final class StringBuffer extends AbstractStringBuilder  
{  
    char value[]; //继承了父类AbstractStringBuilder中的value[]  
    public StringBuffer(String str) {  
         super(str.length() + 16); //继承父类的构造器，并创建一个大小为str.length()+16的value[]数组  
         append(str); //将str切分成字符序列并加入到value[]中  
    }  
}

　　String和StringBuffer内部实现都是一个char数组，但是StringBuffer中的数组分配的空间要比初始化的字符串大16个字符。这样StringBuffer执行append操作时就可以直接加入char数组中。而String却没办法这么做，那么它是如何实现的呢？

　　实际上String执行+操作时，如str1+str2，实际上执行了如下操作： 

StringBuilder temp=new StringBuilder(str1);  
temp.append(str2);  
str=temp.toString();

　　在此过程中，要新创建一个StringBuilder对象和一个String对象，而一旦有大量的String的拼接时，会创建很多StringBuilder对象和一个String对象，这将触发垃圾回收，又会消耗大量时间。因此StringBuffer在执行字符串拼接操作时，要比String效率好很多。

2. StringBuffer与StringBuilder

　　StringBuffer与StringBuilder最大的区别就是线程的安全性。
　　StringBuffer保证了线程的安全，主要通过对关键字设定synchronized属性实现。
　　而StringBuilder无法保障线程的安全，但是在非多线程环境下运行的话，StringBuilder性能好于StringBuffer，因为StringBuilder省去了加锁的过程。

3. 总结

　　字符串不需要经常改变时用String
　　需要经常对字符串进行拼接的情况下优先考虑StringBuilder
　　要在多线程环境下经常对字符串进行拼接时优先考虑StringBuffer
　　另外，如果字符串已经完全确定的情况下，直接使用字符串常量进行+拼接的效率要好于以上三种。比如String s = "I" + " am" + " Lily!";会直接创建一个字符串”I am Lily!”而不是三个，所以它的效率要比之前三个要好。
